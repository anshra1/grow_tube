Dart Class Modifiers
Comprehensive Interview Notes — All 8 Modifiers with Examples
1
class — Standard Class
class Animal {
  String name;
  Animal(this.name);
  void speak() => print('...');
}
  • ✔ Can be instantiated — Animal('Dog')
  • ✔ Can be extended — class Dog extends Animal {}
  • ✔ Can be implemented — class Robot implements Animal {}
  • ✘ Cannot be used with with (it's not a mixin)
No restrictions at all. This is the default class type in Dart.
2
abstract — Blueprint Class
abstract class Shape {
  double area();           // No body → subclass MUST override
  void describe() =>
    print('I am a shape'); // Has body → optional to override
}
  • ✘ Cannot be instantiated — Shape() is a compile-time error
  • ✔ Can be extended — class Circle extends Shape {}
  • ✔ Can be implemented — class Square implements Shape {}
Key Interview Point: Unlike Java, Dart has NO separate interface keyword. Every class implicitly defines an interface. abstract just prevents direct instantiation.
3
base — Extend Only (No Implement)
base class Vehicle {
  void start() => print('Starting...');
}

base class Car extends Vehicle {}   // ✔ Allowed
final class Bike extends Vehicle {} // ✔ Allowed

// ✘ NOT Allowed:
// class Truck implements Vehicle {}  → Cannot implement a 'base' class
// class Van extends Vehicle {}       → Must also be base, sealed, or final
  • ✔ Can be instantiated
  • ✔ Can be extended — but only by base, sealed, or final classes
  • ✘ Cannot be implemented (outside its library)
Why use base? If your class has important logic in its constructor (e.g., initialization, resource management), base forces consumers to use extends so that super logic is always executed. implements would skip all of that.
4
interface — Implement Only (No Extend)
interface class Printable {
  void printData() => print('Printing...');
}

// ✔ Allowed:
class Report implements Printable {
  @override
  void printData() => print('Report data');
}

// ✘ NOT Allowed (outside library):
// class Invoice extends Printable {}
  • ✔ Can be instantiated (unlike abstract)
  • ✔ Can be implemented
  • ✘ Cannot be extended (outside its library)
The OPPOSITE of base. It says: "Use implements only. Don't inherit my implementation." Useful when your implementation details might change in the future.
5
final — Fully Locked Down
final class DatabaseConnection {
  void connect() => print('Connected');
}

// ✘ BOTH are NOT allowed (outside library):
// class MyDb extends DatabaseConnection {}     → Error
// class MockDb implements DatabaseConnection {} → Error
  • ✔ Can be instantiated
  • ✘ Cannot be extended (outside its library)
  • ✘ Cannot be implemented (outside its library)
Think of it as: base + interface locked together = final. Total control.
6
sealed — Exhaustive Pattern Matching
sealed class Result {}

class Success extends Result {
  final String data;
  Success(this.data);
}
class Failure extends Result {
  final String error;
  Failure(this.error);
}

// Compiler knows ALL subtypes → exhaustive switch:
String handle(Result r) => switch (r) {
  Success s => 'Data: ${s.data}',
  Failure f => 'Error: ${f.error}',
  // No default needed!
};
  • ✘ Cannot be instantiated (like abstract)
  • ✔ Can be extended/implemented — but only inside the same library
Key Difference from abstract: sealed restricts subtypes to the SAME library AND enables exhaustive checking. abstract has no such restriction.
7
mixin — Reusable Behavior
mixin Swimming {
  void swim() => print('Swimming!');
}
mixin Flying {
  void fly() => print('Flying!');
}
class Duck extends Animal with Swimming, Flying {}

// Restrict with 'on':
mixin Swimming on Animal {}
// Only classes extending Animal can use this mixin
  • ✘ Cannot be instantiated
  • ✘ Cannot be extended
  • ✔ Applied using with
  • ✔ Can use on keyword to restrict to specific superclasses
⚡ Mixin Linearization Order (MRO) — Very Important!

class Impl extends Base with Mixin1, Mixin2

The super chain is: Impl → Mixin2 → Mixin1 → Base
• super in Impl calls Mixin2 (the last mixin applied)
• super in Mixin2 calls Mixin1
• super in Mixin1 calls Base

The RIGHTMOST mixin's method wins if multiple mixins define the same method.
8
mixin class — Class + Mixin Hybrid
mixin class Logger {
  void log(String msg) => print('[LOG] $msg');
}

class Service with Logger {}        // ✔ Use as a mixin
class FileLogger extends Logger {}  // ✔ Use as a class
var l = Logger();                   // ✔ Instantiate directly
  • ✔ Can be instantiated
  • ✔ Can be extended
  • ✔ Can be used with with
  • ✘ Cannot use on keyword (unlike a regular mixin)
Maximum flexibility — acts as BOTH a class AND a mixin.

🎯 Top Interview Questions & Answers

Q1: What's the difference between base and interface?
base allows extends only (blocks implements). interface allows implements only (blocks extends). They are exact opposites.
Q2: Why would you use sealed?
For exhaustive pattern matching with switch. The compiler guarantees you've handled every possible subtype. No default case needed.
Q3: What is Mixin Linearization Order (MRO)?
Mixins are applied left-to-right, so the super chain goes right-to-left. In with M1, M2, super calls go: M2 → M1 → Base. The rightmost mixin wins.
Q4: Can you combine base and interface?
No, they are contradictory. If you want to block both extends AND implements, use final instead.
Q5: What's the difference between abstract and sealed?
sealed restricts subtypes to the same library AND enables exhaustive checking. abstract has no restriction on where subtypes can be created.
Q6: What's the difference between mixin and mixin class?
mixin can ONLY be used with with. mixin class can be used with with, extends, AND instantiated directly. mixin can use on; mixin class cannot.
Q7: Why can't a regular class be used as a mixin?
Dart separates the concept of classes and mixins. If you want both behaviors, use mixin class.
Q8: What happens when two mixins define the same method?
The last mixin in the with clause wins. Its method is called first in the super chain.
